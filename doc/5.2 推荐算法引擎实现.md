### 5.2 推荐算法引擎实现

推荐引擎是本系统的核心大脑，它整合了大规模分布式离线计算、轻量级在线实时反馈以及前沿的自然语言认知技术，构成了一个三级混合推荐架构。本节将详细阐述这三个层级的算法引擎在工程上的具体实现过程。

#### 5.2.1 Spark 离线模型训练与结果持久化

为了突破单机内存在处理千万级历史评分矩阵时的瓶颈，系统采用 Apache Spark 框架实现了交替最小二乘法（ALS）模型的分布式训练。该任务被封装在独立的 `spark_runner.py` 脚本中运行。

1. **Spark 环境初始化与数据加载**：脚本首先初始化一个本地模式（`local[*]`）的 `SparkSession`，并为其 Driver 进程分配了充足的堆内存（如 `2g`）以应对数据序列化开销。随后，利用 SQLAlchemy 与 Pandas 从底层的 PostgreSQL 数据库中全量读取 `user_personal_ratings` 表的数据（提取 `user_id`, `tconst`, `rating`），并将其转化为分布式的 Spark DataFrame。
2. **标识符的数值化编码**：由于 Spark MLlib 的 ALS 算法强制要求用户 ID 和物品 ID 必须是整型数值，而 IMDb 的影视主键（`tconst`）是类似 `tt0000001` 的字符串。因此，系统在将数据送入模型前，引入了 `StringIndexer` 特征转换算子。该算子在分布式环境下对 `tconst` 列进行扫描计数，并将其映射为连续的整型索引列 `movie_id_int`。
3. **分布式模型拟合与预测**：系统实例化 `ALS` 估计器，配置了关键的超参数：最大迭代次数 `maxIter=10`，正则化参数 `regParam=0.1` 以防止过拟合，并开启了非负矩阵分解模式（`nonnegative=True`）。同时，设置 `coldStartStrategy="drop"` 策略，确保在遇到未见过的测试集数据时丢弃产生 NaN 预测值的记录。调用 `.fit(ratings_df)` 触发集群内的交替寻优计算。模型收敛后，直接调用 `.recommendForAllUsers(10)` 为全库每一位活跃用户生成得分最高的 10 部影视推荐。
4. **数据平铺与逆向解码落盘**：生成的推荐结果采用了高度压缩的嵌套数组格式（即一条用户记录对应一个包含多部电影推荐的数组）。系统利用 Spark SQL 的 `explode` 函数将其“炸开”为平铺的关系型结构。接着，利用 `IndexToString` 算子将内部使用的 `movie_id_int` 重新转换回标准的 IMDb `tconst` 字符串主键。最后，使用 SQLAlchemy 驱动先执行 `TRUNCATE TABLE spark_recommendations` 清空旧批次数据，再将转换后的 Pandas DataFrame 通过 `to_sql` 方法全量覆写至 PostgreSQL，完成离线计算闭环。

#### 5.2.2 基于内存矩阵的 CF 实时推断实现

为了捕捉用户的即时兴趣变化，系统在 Web 后端服务层（`services/recommendation_service.py`）实现了一套基于物品的协同过滤（Item-CF）在线推荐引擎。为了保证推断的时效性，该引擎采用了“重构离线相似度矩阵，轻量化在线加权计算”的工程架构。

1. **多模态特征融合与矩阵构建**：在后台管理员触发的 `train_model()` 任务中，系统首先异步抽取数据库中的显式评分数据（`user_personal_ratings`），同时将用户的隐式收藏行为（`user_favorites`）等效转化为 10 分的极高权重，融合进训练数据集中，以此有效缓解数据的稀疏性。利用 Pandas 的 `pivot` 函数生成以 `tconst` 为行、`user_id` 为列的用户-物品评分透视表（Pivot Table），缺失值填充为 0。
2. **高维空间余弦相似度计算与序列化**：将透视表转换为稀疏矩阵后，调用 Scikit-learn 的 `cosine_similarity` 函数，在内存中计算出所有影视作品之间的相似度矩阵。为了避免每次服务器重启都重新计算，系统使用 Python 的 `pickle` 模块将该高维矩阵物理化存储至本地磁盘的 `data/recommendation_model.pkl` 文件中。在系统启动时，通过 `load_model()` 方法将其一次性加载并常驻内存（单例模式 `_similarity_df`）。
3. **毫秒级在线推荐打分**：当接收到前台请求 `get_recommendations()` 时，系统首先查询该用户历史交互过的“种子影片”集合（包括收藏和高分评价的电影）及其对应的权重得分。随后，算法在内存中快速遍历这些种子影片，从全局相似度矩阵 `_similarity_df` 中提取与之相似的候选影片（剔除相似度低于 0.1 的长尾噪音）。候选影片的最终推荐分数为其与所有种子影片相似度的加权求和（公式：`score = similarity * weight`）。最后，对结果进行降序排列，剔除用户已看过的影片，截取 Top N 并在底层宽表中拉取海报和详情返回前端，实现了端到端的快速闭环。

#### 5.2.3 “情感树洞”自然语言解析与推荐下发

系统的认知与情感交互层部署在 `services/analysis_service.py` 模块中。该模块打破了传统的离散关键字检索，实现了对用户复杂长文本情绪意图的精准捕捉与场景映射。

1. **预训练语言模型的单例加载与零样本推断**：为了兼顾中文语境下的语义理解精度与推理速度，系统选用了 Hugging Face 开源的 `chinese_xlm_xnli` 跨语言大模型。为了避免频繁读写磁盘导致的 I/O 阻塞，在服务层启动时，使用 `@functools.lru_cache(maxsize=1)` 装饰器将 `pipeline("zero-shot-classification")` 实例缓存在服务器内存中。在处理前端传来的文本（如“今天好累，不想动脑子”）时，`analyze_text_mood` 函数将系统预设的七大情绪场景名称（如“开心”、“难过”、“烧脑”）作为候选标签空间（Candidate Labels）一同送入模型。模型通过计算假设蕴含关系，输出文本归属于各情绪标签的置信度。系统设定阈值 $0.3$，仅保留置信度最高且有效的核心情绪标签。
2. **心理学映射引擎与智能分发**：成功捕获用户情绪（例如“难过”）后，`get_movies_by_mood` 函数会触发系统的启发式映射字典 `MOOD_SCENARIOS`。该字典将抽象的情绪状态转化为具象的数据库查询策略：例如针对“难过”，系统不仅会匹配“Drama”和“Romance”等共情题材，还会混合“Family”和“Animation”等具有治愈属性的类目（`target_genres = primary_genres + healing_genres`）。系统随后利用 SQLAlchemy 的 `or_` 操作符动态组装 SQL 查询，并强制附加 `numVotes > 5000`（高投票数基数保障）以确保推荐影片的优质性。
3. **“偶遇感”结果排序**：为防止在相同情绪下总是推荐固定的高分老片导致用户审美疲劳，系统在获取 50 部高评分候选集后，在内存中使用 `random.shuffle()` 算法将结果集打乱，再截取前 12 部，赋予每次情感交互一定的“盲盒感”与随机性。最终，生成的推荐片单会连带预设的暖心文案（如“🌻 抱抱你。哭出来没关系，让这些故事陪陪你...”），通过前后端状态同步无缝呈现给用户，完成了高度人性化的推荐交互。
