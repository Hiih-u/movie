### 5.1.1 离线 IMDb 数据集清洗与高速导入

本系统底层的核心影视实体数据、演职人员数据以及全局评分数据，均直接来源于 IMDb 官方提供的海量开放数据集。该数据集以 GZIP 压缩的 TSV（Tab-Separated Values）格式分发，数据量达数 GB，包含数千万条记录。为了将这些原始数据高效、准确地转换为推荐系统可用的结构化底层知识库，系统设计了一套包含解压、结构定义、高速批量入库及后置索引优化的数据预处理流水线。

**1. 数据获取与解压缩**
首先，通过 Linux 环境下的 shell 脚本批量获取原始数据，并使用 `gunzip *.tsv.gz` 命令对所有压缩文件进行解压，得到原生的 TSV 文本文件。

**2. 核心底表结构定义 (DDL)**
在 PostgreSQL 数据库中，预先定义了接收原始数据的物理表结构。为了最大程度兼容 IMDb 原始数据的扁平化格式，系统在初始导入阶段将包含多个元素的字段（如 `title_basics` 表中的 `genres` 题材分类、`name_basics` 表中的 `knownForTitles` 代表作）统一定义为 `TEXT` 类型（逗号分隔的字符串），以便后续进行平滑的数据清洗与转换。核心构建的表包括影视基础表（`title_basics`）、演职员表（`name_basics`）、剧组关联表（`title_crew`）、剧集表（`title_episode`）以及全局评分表（`title_ratings`）。

**3. 基于 \COPY 命令的高速清洗与导入机制**
面对千万级别的海量记录，传统的按行执行 `INSERT` SQL 语句会导致灾难性的网络延迟与磁盘 I/O 阻塞，耗时可能长达数天。为此，本系统利用 PostgreSQL 原生提供的高效数据加载工具 `psql` 与 `\COPY` 命令，将磁盘文本文件直接流式写入数据库引擎。
在导入的过程中，系统同时完成了关键的“脏数据清洗”步骤：

* **空值转换**：IMDb 数据集中使用特定的字符 `\N` 来表示缺失值。通过在导入命令中显式声明 `NULL '\N'`，系统指令 PostgreSQL 在解析数据流时，自动将所有的 `\N` 文本精准转换为关系型数据库标准的 `NULL` 值，保证了数据完整性约束的严谨性。
* **定界与转义**：指定 `DELIMITER E'\t'` 按照制表符切分字段，并使用 `QUOTE E'\b'` （退格符）作为引用字符，以防原始文本中出现的常规引号破坏 TSV 的列对齐规则。

导入脚本示例代码如下：

```sql
\COPY title_basics FROM '/home/wly/movie_data/title.basics.tsv'
WITH (FORMAT csv, DELIMITER E'\t', NULL '\N', HEADER true, QUOTE E'\b');

```

**4. 后置索引建立与优化**
在数据库工程实践中，如果在空表上先建立索引再大批量导入数据，每一行数据的插入都会触发索引树的重构，从而严重拖慢导入速度。因此，本系统采取了**“先导数据，后建索引”**的优化策略。
在所有海量数据通过 `\COPY` 成功落盘后，系统才执行相关的索引创建脚本。例如，针对 `title_ratings` 表的投票数字段建立 B-Tree 索引（`CREATE INDEX idx_ratings_votes ON title_ratings(numVotes)`），以及针对 `title_episode` 表的父剧集字段建立外键关联索引（`CREATE INDEX idx_episode_parent ON title_episode(parentTconst)`），以此在保证极速初始化的同时，兼顾了线上业务的高并发查询性能。
