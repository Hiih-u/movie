### 2.2.2 Apache Spark 与分布式大数据处理

在本影视推荐系统中，随着接入的 IMDb 官方数据集（包含数百万部影视作品、千万级的演职人员关系以及海量的用户交互评分数据）规模的不断扩大，单机环境下的数据处理与机器学习模型训练面临着严峻的性能瓶颈与内存溢出（OOM）风险。为了实现高效的离线推荐计算，本系统引入了 **Apache Spark** 作为分布式大数据处理引擎。

**1. 分布式计算的需求背景**
传统的 Python 数据科学栈（如 Pandas、Scikit-learn）通常将会话数据完全加载至单机内存中进行单线程或多线程计算。当面对千万级别的“用户-影视”评分矩阵时，矩阵的维度极高且极其稀疏。如果在单机内存中直接构建并分解这样庞大的稀疏矩阵，不仅计算耗时难以忍受，对服务器内存的需求也是灾难性的。因此，引入能够将计算任务切分并调度至多个工作节点（Worker Nodes）并发执行的分布式计算框架成为了必然选择。

**2. Apache Spark 的核心优势与 PySpark 架构**
Apache Spark 是专为大规模数据处理而设计的快速、通用的计算引擎。相较于早期的 Hadoop MapReduce，Spark 的核心优势在于其**基于内存的计算（In-Memory Computing）**机制。它通过引入弹性分布式数据集（Resilient Distributed Dataset, RDD）和 DataFrame 数据抽象，允许中间计算结果缓存在内存中，极大地减少了磁盘 I/O 开销，使得迭代式机器学习算法（如矩阵分解）的执行速度获得了指数级提升。
在本课题的工程实现中，系统采用了 **PySpark** 接口。PySpark 借助 Py4J 库在 Python 解释器与运行在 Java 虚拟机（JVM）上的 Spark Core 之间建立了通信桥梁。这种架构既保留了 Python 在数据逻辑编写上的简洁性，又完美继承了 Spark 在 JVM 底层的强大分布式调度与计算能力（本系统明确要求配置 JDK 17 以保障底层的最佳性能与兼容性）。

**3. Spark 在本系统中的具体应用流水线**
在本系统（MovieRec Sys）中，Spark 主要承担**离线推荐模型训练**与**全局候选集生成**的重任。具体的处理流水线如下：

* **数据加载与预处理**：在离线计算脚本中，Spark 引擎从 PostgreSQL 数据库中批量抽取用户的海量历史评分数据（`user_personal_ratings` 表）。由于 IMDb 的影视主键（`tconst`，如 `tt0000001`）为字符串格式，而协同过滤算法通常要求输入为数值型矩阵索引，系统利用 PySpark MLlib 提供的 `StringIndexer` 特征转换器，在分布式环境下高效地将千万级字符串 ID 映射为连续的整型 ID，从而完成了特征的数值化预处理。
* **ALS 矩阵分解与模型拟合**：预处理后的 DataFrame 会被送入 Spark MLlib 的交替最小二乘法（Alternating Least Squares, ALS）模型中进行训练。Spark 会将庞大的用户-物品评分矩阵自动切分，并在多个 Executor 上并发执行最小二乘优化，通过迭代逼近的方式挖掘出用户和物品的隐含特征向量（Latent Vectors），有效克服了传统协同过滤面临的数据稀疏性难题。
* **Top-N 推荐生成与回写**：模型训练完成后，调用 `recommendForAllUsers` 方法为全库所有用户并行计算出预测评分最高的 Top-N（本系统设定为 10 部）推荐列表。由于生成的原始数据包含了嵌套的数组结构，系统进一步利用 Spark SQL 的 `explode` 扁平化函数与 `IndexToString` 逆向转换器，将整型 ID 还原为原始的 `tconst` 字符串，最后将数以千万计的推荐结果批量回写至关系型数据库的 `spark_recommendations` 表中。

通过集成 Apache Spark，本系统彻底摆脱了单机内存与算力的桎梏，成功构建了能够支撑千万级数据规模的离线推荐流水线，为上层应用提供了高可用、广覆盖的个性化推荐底座。
