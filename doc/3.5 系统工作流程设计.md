### 3.5 系统工作流程设计

系统架构与功能模块界定了系统的静态结构，而工作流程设计则描述了系统在实际运行时的动态行为。为了清晰地展示系统的内部流转机制，本节将详细梳理系统中最核心的四大业务工作流：用户注册与鉴权流程、混合与情感推荐获取流程、用户交互评分流程，以及后台离线计算与数据看板更新流程。

#### 3.5.1 用户注册与鉴权流程

用户账号体系是实现个性化推荐的基石。本系统的注册与鉴权流程严格遵循信息安全规范，其具体步骤如下：

1. **信息提交**：用户在前端注册页面（`/register`）填写用户名、密码，并可选填性别、年龄段等基础画像信息。
2. **唯一性校验**：前端提交请求后，后端 `auth_service.py` 首先通过 SQLAlchemy 异步查询 `users` 表，校验该用户名是否已被注册。若存在冲突，则通过 NiceGUI 的 `ui.notify` 抛出错误提示。
3. **密码哈希加密**：校验通过后，系统调用 `passlib` 库的 `bcrypt` 算法，对用户输入的明文密码进行加盐哈希（Hash）处理，确保数据库中不留存明文密码。
4. **持久化与会话建立**：将包含哈希密码及默认角色（`role='user'`）的用户实体对象写入 PostgreSQL 数据库。注册成功后，用户跳转至登录页。登录验证成功时，系统将用户的 `user_id`、`username` 及权限标识写入框架的全局会话字典 `app.storage.user` 中，完成鉴权上下文的初始化。

#### 3.5.2 多模态推荐获取流程

系统为用户提供推荐内容的过程是整个业务的核心，根据触发场景的不同，分为“常规混合推荐”与“NLP 情感推荐”两条工作流：

**1. 常规混合推荐工作流（首页展示）**

* **步骤一（冷启动检测）**：用户进入首页，后端识别其身份。如果用户是刚注册的新用户，数据库中无任何交互记录（即 `user_personal_ratings` 为空），系统触发“冷启动策略”，直接从底层数据库拉取按全网平均分（`averageRating`）和投票数（`numVotes`）加权排序的“经典必看榜单”返回给前端。
* **步骤二（离线召回）**：若为老用户，系统首先查询 `spark_recommendations` 表，快速提取由 Apache Spark ALS 模型在离线阶段为其计算好的长线潜在偏好 Top-N 影片 ID。
* **步骤三（在线排序）**：随后，系统调用内存中的 `recommendation_service.py`（Item-CF 算法）。提取该用户最近的收藏和打分记录作为上下文权重，与离线候选集进行相似度矩阵的二次点积运算，完成微调打分。
* **步骤四（数据封装与渲染）**：后端根据最终排好序的影片 ID，从缓存库或主表中提取电影的元数据（海报、标题、评分），返回给前端进行瀑布流渲染。

**2. 情感树洞推荐工作流（自然语言触发）**

* **步骤一（语义输入）**：用户在前端“情感树洞”弹窗中输入一段非结构化的情绪文本。
* **步骤二（零样本分类推断）**：请求发送至后端 `analysis_service.py`。系统将文本输入 Transformer 大模型（`chinese_xlm_xnli`），计算其与预设情绪标签库（如“难过”、“刺激”、“疲惫”）的匹配逻辑值，并提取出置信度得分最高且大于阈值（0.3）的情绪标签。
* **步骤三（启发式策略映射）**：根据捕获的标签，查表获取对应的观影策略。例如，“疲惫”被映射为“喜剧/动画题材 + 评分大于 8.0”。
* **步骤四（动态查询与反馈）**：通过 SQLAlchemy 动态拼接 SQL 执行查询。系统将查询到的高分治愈影片连同预设的“暖心心理学文案”一并推送到前端，完成一次充满人文关怀的推荐闭环。

#### 3.5.3 交互反馈与评分闭环流程

推荐系统是一个不断进化的反馈闭环（Feedback Loop），用户对推荐结果的交互将直接影响下一次的推荐质量。交互流程如下：

1. **动作触发**：用户在浏览电影卡片或详情页时，点击“红心”进行收藏，或拖动滑动条进行 1-10 分的评分操作。
2. **异步入库（Upsert 操作）**：前端通过 WebSocket 将交互指令与 `tconst`（影片 ID）发送至后端 `interaction_service.py`。后端采用 Upsert（存在则更新，不存在则插入）逻辑，安全地将该条交互记录持久化至 `user_favorites` 或 `user_personal_ratings` 关系表中。
3. **实时模型状态更新**：记录入库后，不仅用户的个人中心列表会实时刷新，更重要的是，系统内存中维护的该用户“即时行为特征向量”被瞬间修改。
4. **即时反馈（闭环）**：当用户刷新页面或请求下一页推荐时，在线的 Item-CF 模型将读取这部分刚刚产生的增量特征权重。由于新收藏的影片相似度被放大，系统会立刻在推荐列表中上调与其相似的电影排名，实现了“所点即所得”的敏捷反馈。

#### 3.5.4 离线计算与数据看板更新流程

为了不阻塞核心的 Web 交易流程，海量数据的计算与图表生成被设计为独立的后台异步/离线工作流，由超级管理员统筹调度：

1. **算法重训与缓存刷新流**：
* 管理员在后台仪表盘点击“重新训练模型”或“执行 Spark 离线计算”。
* 系统挂起后台子进程：如果是 Item-CF，则利用 Pandas 提取全量最新交互数据，调用 Scikit-learn 重新计算相似度矩阵并覆盖内存对象；如果是 Spark ALS，则启动 PySpark 集群资源，执行分布式的矩阵分解计算，最终将结果批量覆盖回 PostgreSQL。
* 计算完成后，系统通过 WebSocket 向管理员的浏览器主动推送“模型已就绪”的成功通知。


2. **商业大屏数据渲染流**：
* 鉴于千万级全域数据分析的极高耗时，数据大屏并非在管理员每次访问时实时渲染。
* 管理员需手动（或通过系统 Cron 定时任务）触发 `generate_charts.py` 脚本。
* 该脚本利用 Pandas 从数据库中进行复杂的分组、聚合与透视计算（如统计各年代的题材分布）。随后，调用 Plotly 引擎将这些 DataFrame 结构转化为具有高度交互性的 HTML 静态视图文件。
* 新生成的 `.html` 文件会覆盖服务器静态目录中的旧文件。当管理员访问 `/admin/analytics` 路由时，前端只需通过轻量级的 `iframe` 组件加载这些本地静态资源，即可获得毫无卡顿的极致数据大屏体验。
