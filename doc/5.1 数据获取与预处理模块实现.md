### 第五章 系统的详细设计与实现

本章将结合前文的需求分析与架构设计，详细阐述影视推荐系统（MovieRec Sys）各个核心功能模块的编码实现过程。重点介绍数据获取与预处理、混合推荐算法的工程落地、NLP 情感交互模块的集成，以及前端 UI 与后台管理可视化大屏的构建细节。

#### 5.1 数据获取与预处理模块实现

高质量、多维度的数据是推荐系统发挥价值的先决条件。本系统的底层数据源不仅包含海量的结构化静态文件，还依赖于动态的网络爬虫与第三方 API。因此，数据获取与预处理模块（ETL）的实现是系统开发的第一步。

**5.1.1 IMDb 基础数据的离线清洗与导入**

系统的基础影视知识库（涵盖百万级电影/剧集、千万级演职人员及评分）直接来源于 IMDb 官方提供的离线 TSV（Tab-Separated Values）数据集。

1. **数据清洗**：原始 TSV 文件中存在大量用 `\N` 表示的缺失值，以及不规范的数组格式（如逗号分隔的分类标签）。在导入前，系统利用 Python 脚本对这些脏数据进行了清洗。例如，针对 `title.basics.tsv` 中的 `genres` 字段，脚本将其按逗号切分，并过滤掉空值，转换为标准的 Python List 结构，以便后续映射为 PostgreSQL 的 `ARRAY` 类型。
2. **批量入库优化**：由于数据量极其庞大（数 GB），若采用传统的逐行 `INSERT` 语句会导致极高的网络与磁盘 I/O 开销。系统采用了 PostgreSQL 原生的批量复制命令（如 `COPY FROM`）或利用 SQLAlchemy 提供的 `insert().values()` 配合批处理（Batch Size = 10000）机制，大幅缩短了初始化时间，成功构建了 `title_basics`、`name_basics`、`title_ratings` 等核心底层关系表。

**5.1.2 基于异步协程的外部数据爬虫实现**

为了弥补 IMDb 官方数据在“商业票房指标”与“中文本地化评价”上的缺失，系统利用 Python 实现了高并发的异步网络爬虫，动态抓取 Box Office Mojo 与豆瓣的数据。

1. **异步并发架构**：在 `crawlers/box_office_crawler.py` 和 `crawlers/top250_crawler.py` 中，系统摒弃了传统的同步 `requests` 库，全面采用了 `asyncio` 配合异步 HTTP 客户端 `httpx`。这种非阻塞 I/O 架构允许爬虫在等待网络响应时主动挂起当前协程，去发起下一个请求，从而在单线程下实现了极高的抓取吞吐量。
2. **DOM 解析与数据提取**：通过引入 `BeautifulSoup4` 库，系统精准定位目标网页的 HTML DOM 节点。例如，在豆瓣 Top 250 爬虫中，系统通过 CSS 选择器 `.grid_view li` 遍历每一部电影，提取中文译名、豆瓣评分（`rating_num`）、评价人数以及隐藏在链接底部的 IMDb ID，为后续中西审美差异对比（双向柱状图）提供了关键数据支撑。
3. **Upsert 数据同步机制**：为了保证数据的幂等性，爬取到的数据在持久化至 `douban_top250` 和 `movie_box_office` 表时，采用了 PostgreSQL 的 `ON CONFLICT DO UPDATE`（即 Upsert）语法。若数据库中不存在该记录则插入，若存在则更新票房或评分数值，有效避免了主键冲突。

**5.1.3 TMDB 海报资源的动态同步与缓存**

影视海报是前端展示的重要元素，但 IMDb 官方数据集并未提供合法的图片 URL。系统通过接入 The Movie Database (TMDB) API 来解决这一问题。

为了避免在离线阶段对千万级影视库进行全量 API 请求（会导致封停和巨大的时间成本），系统在 `services/tmdb_service.py` 中实现了**“按需获取与持久化缓存”**策略：

1. **拦截与校验**：当前端页面需要渲染某部电影（如首页推荐流或电影详情页）时，后端首先校验该电影记录中的 `poster_url` 字段是否为空。
2. **异步外部调用**：若为空，系统立即触发异步函数，携带该电影的 `tconst`（IMDb ID）作为参数，向 TMDB API 发起 GET 请求，解析返回的 JSON 数据获取高质量的海报路径。
3. **回写与缓存**：获取到海报链接后，系统不仅将其返回给前端用于即时渲染，还会立即执行异步数据库更新操作，将该 URL 写入 `title_basics` 和 `movie_summary` 表中。当该影片下次被任何用户访问时，系统将直接从本地数据库读取缓存的海报链接，彻底消除了重复的外部网络开销。

**5.1.4 数据降维与前端缓存表构建**

在千万级的数据量下，如果前端每一次分页或检索都需要对 `title_basics`（基础信息表）与 `title_ratings`（全局评分表）进行复杂的 `JOIN` 联表查询和 `ORDER BY` 排序，将导致严重的查询阻塞，无法满足 Web 毫秒级响应的需求。

为此，系统在预处理阶段引入了**反范式化（Denormalization）设计**，专门构建了一张面向前端展示的扁平化宽表——`movie_summary`。

1. **数据聚合**：通过执行初始化脚本（如 `init/init_db.py` 中的构建逻辑），系统将 `title_basics` 中的标题、年份、题材数组，与 `title_ratings` 中的平均分、投票数聚合在同一张表中。
2. **过滤与截断**：为了保证推荐池的质量，聚合过程中系统直接过滤掉了投票数极低（如少于 100 票）的冷门低质影片以及短片（Shorts），大幅缩减了在线检索的数据规模。
3. **索引优化**：为该宽表的关键检索字段（如 `primaryTitle`, `averageRating`）建立了 B-Tree 索引，并为数组类型的 `genres` 字段构建了 GIN（Generalized Inverted Index）索引，使得前端的分类筛选与模糊搜索性能提升了数个数量级。
