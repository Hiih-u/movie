# 整体架构设计说明

## 一、架构设计思路

本项目采用 **现代化全栈架构**，基于 **Python 3.11** 生态，整合了 **Web 开发**、**大数据处理**、**机器学习** 和 **数据可视化** 技术。架构设计的核心原则是：

1. **技术栈统一性**：尽可能使用 Python 生态，降低学习成本和维护复杂度
2. **性能优先**：异步架构 + 高性能组件，支持高并发和大数据处理
3. **开发效率**：全栈开发 + 丰富组件库，快速迭代和原型验证
4. **可扩展性**：模块化设计，便于功能扩展和技术升级

## 三、分层架构详解

### 1. 用户界面层（NiceGUI）

**为什么选择 NiceGUI？**

**传统前后端分离的问题**
```
前端：Vue 3 + TypeScript + Vite
后端：FastAPI + SQLAlchemy
问题：
- 需要学习两套技术栈
- 前后端接口联调成本高
- 数据序列化/反序列化开销
- 部署配置复杂
```

**NiceGUI 的优势**
```
全栈：Python (NiceGUI)
优势：
- 单一语言开发，降低学习成本
- 无需前后端接口联调
- 数据直接渲染，无序列化开销
- 单一部署，配置简单
```

**项目应用**
- 15 个页面全部使用 Python 编写
- 50+ UI 组件，丰富的交互体验
- WebSocket 实时通信，无刷新更新
- 响应式设计，移动端友好

### 2. Web 框架层（FastAPI）

**为什么选择 FastAPI？**

**性能需求**
```
项目需求：
- 千万级 IMDb 数据查询
- 高并发用户访问（浏览、收藏、评分）
- 实时推荐算法响应（毫秒级）
- 数据可视化大屏的复杂查询

FastAPI 优势：
- 原生异步支持（async/await）
- 性能接近 Node.js 和 Go
- 比 Flask 快 2-3 倍
- 比 Django 快 5-10 倍
```

**异步架构的重要性**
```python
# 同步框架（Flask/Django）
def get_movies():
    # 阻塞等待数据库响应
    movies = db.query("SELECT * FROM movies")
    return movies

# 异步框架（FastAPI）
async def get_movies():
    # 非阻塞等待，可以处理其他请求
    movies = await db.execute("SELECT * FROM movies")
    return movies
```

**并发能力对比**
- 同步框架：100 个并发请求需要 100 个线程
- 异步框架：100 个并发请求只需要少量线程
- 资源占用更少，并发能力更强

**项目应用**
- 所有服务层都是异步函数（50+ async 函数）
- 异步数据库操作（asyncpg）
- 异步 HTTP 请求（TMDB API、爬虫）
- 异步页面交互（用户操作无阻塞）

### 3. 业务逻辑层（Services）

**为什么分层设计？**

**单一职责原则**
```
每个服务负责一个业务领域：
- recommendation_service.py：推荐算法
- analysis_service.py：数据分析 + NLP
- movie_service.py：电影业务逻辑
- interaction_service.py：用户交互
- user_service.py：用户管理
- auth_service.py：认证授权
```

**优势**
- 代码组织清晰，易于维护
- 职责分离，降低耦合
- 便于单元测试
- 支持团队协作

**项目应用**
- 11 个服务模块，50+ 业务函数
- 每个服务独立，互不干扰
- 统一的异步接口设计
- 便于功能扩展

### 4. 数据访问层（SQLAlchemy）

**为什么选择 SQLAlchemy？**

**ORM 的优势**
```python
# 原生 SQL
cursor.execute("SELECT * FROM users WHERE username = ?", (username,))
user = cursor.fetchone()

# SQLAlchemy ORM
user = session.query(User).filter(User.username == username).first()
```

**SQLAlchemy 的优势**
- 类型安全，编译时检查
- 自动处理 SQL 注入
- 数据库无关（支持多种数据库）
- 关系映射（外键、关联）
- 迁移工具（Alembic）

**为什么选择 SQLAlchemy 2.0？**
```python
# SQLAlchemy 1.x（同步）
session = Session()
users = session.query(User).all()

# SQLAlchemy 2.0（异步）
async with AsyncSessionLocal() as session:
    users = (await session.execute(select(User))).scalars().all()
```

**优势**
- 原生异步支持
- 现代化的查询语法
- 更好的类型提示
- 与 FastAPI 完美配合

**项目应用**
- 12 个数据模型（models.py）
- 复杂的关联查询（JOIN）
- 事务管理（ACID）
- 索引优化

### 5. 数据存储层（PostgreSQL）

**为什么选择 PostgreSQL？**

**性能优势**
```
项目需求：
- 千万级 IMDb 数据存储
- 复杂的多表关联查询
- 高并发读写操作
- 数据分析统计

PostgreSQL 优势：
- MVCC 并发控制，读写互不阻塞
- 复杂查询优化器
- 支持部分索引、表达式索引
- 大数据量处理能力强
```

**功能优势**
```
高级特性：
- JSON/JSONB 支持（存储扩展信息）
- 数组类型（存储电影类型列表）
- 全文搜索（电影标题搜索）
- 窗口函数（排名分析）
- CTE（复杂查询）
```

**双驱动支持**
```python
# Web 服务：asyncpg（异步）
DATABASE_URL = "postgresql+asyncpg://user:pass@localhost/db"
engine = create_async_engine(DATABASE_URL)

# Spark 任务：psycopg2（同步）
DATABASE_URL = "postgresql+psycopg2://user:pass@localhost/db"
engine = create_engine(DATABASE_URL)
```

**优势**
- asyncpg：高性能异步驱动，匹配 FastAPI
- psycopg2：成熟的同步驱动，匹配 Spark
- 灵活适配不同场景

**项目应用**
- 千万级 IMDb 数据导入（COPY 命令）
- 复杂的多表关联查询
- 高并发用户操作
- 数据分析统计

### 6. 大数据处理层（Spark）

**项目应用**

### 3. 事务管理

**ACID 保证**
```python
async def toggle_favorite(user_id, tconst):
    async with AsyncSessionLocal() as db:
        try:
            # 开始事务
            # 检查是否已收藏
            existing = await db.execute(
                select(UserFavorite).where(
                    UserFavorite.user_id == user_id,
                    UserFavorite.tconst == tconst
                )
            )
            
            if existing.scalar():
                # 删除收藏
                await db.execute(
                    delete(UserFavorite).where(...)
                )
            else:
                # 添加收藏
                db.add(UserFavorite(...))
            
            # 提交事务
            await db.commit()
        except Exception as e:
            # 回滚事务
            await db.rollback()
            raise e
```

**优势**
- 数据一致性
- 原子性操作
- 错误恢复
- 并发安全
