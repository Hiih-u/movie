# 数据库选型说明：PostgreSQL vs MySQL

## 一、选型结论

本项目选择 **PostgreSQL** 作为数据库，而非 MySQL。这一选择基于项目的技术需求、性能要求和长期维护考虑。

## 二、PostgreSQL 的核心优势

### 1. 更强大的数据类型支持

**JSON/JSONB 类型**
- PostgreSQL 原生支持 JSON/JSONB，查询性能优秀
- 本项目中可用于存储电影的扩展信息（如多语言标题、元数据）
- 支持索引和复杂查询，而 MySQL 的 JSON 支持相对较弱

**数组类型**
- PostgreSQL 支持数组字段，适合存储电影类型列表（如 `['Action', 'Adventure']`）
- 避免了需要关联表或字符串分割的复杂性
- MySQL 不支持原生数组类型

**全文搜索**
- PostgreSQL 内置强大的全文搜索功能
- 可用于电影标题、剧情简介的模糊搜索
- 支持中文分词（需配置 zhparser 扩展）

### 2. 更优秀的并发控制

**MVCC（多版本并发控制）**
- PostgreSQL 的 MVCC 实现更成熟，读写互不阻塞
- 本项目有大量并发用户操作（收藏、评分、浏览）
- 避免了 MySQL InnoDB 的间隙锁问题

**连接池性能**
- PostgreSQL 在高并发场景下表现更稳定
- 本项目使用 asyncpg 异步驱动，充分利用 PostgreSQL 的并发优势

### 3. 更复杂查询的性能优势

**窗口函数**
- PostgreSQL 对窗口函数支持更完善
- 本项目数据分析中需要使用 RANK、ROW_NUMBER 等函数
- 例如：计算电影排名、用户活跃度排行

**CTE（公用表表达式）**
- PostgreSQL 对递归 CTE 支持更好
- 可用于复杂的层级查询（如剧集的父子关系）

**索引优化**
- PostgreSQL 支持部分索引、表达式索引
- 例如：只为评分 > 7.0 的电影创建索引
- 本项目在 `init/add_index.py` 中使用了大量索引优化

### 4. 更好的数据完整性

**外键约束**
- PostgreSQL 的外键检查更严格
- 确保数据一致性（如用户删除时级联处理收藏记录）

**CHECK 约束**
- 支持复杂的 CHECK 约束
- 例如：评分必须在 1-10 之间，年份必须合理

### 5. 更适合大数据场景

**表分区**
- PostgreSQL 支持表分区，适合海量数据
- 本项目有千万级 IMDb 数据，可按年份分区优化查询

**并行查询**
- PostgreSQL 支持并行查询执行
- 加速复杂分析查询（如数据可视化大屏的统计查询）

## 三、本项目中的具体应用

### 1. 异步驱动支持

```python
# Web 服务使用 asyncpg（高性能异步驱动）
DATABASE_URL = "postgresql+asyncpg://postgresuser:password@localhost:5432/movie_db"

# Spark 任务使用 psycopg2（同步驱动）
DATABASE_URL = "postgresql+psycopg2://postgresuser:password@localhost:5432/movie_db"
```

PostgreSQL 的 asyncpg 驱动性能远超 MySQL 的异步驱动，适合 FastAPI 异步架构。

### 2. 复杂关联查询

本项目数据模型复杂，涉及多表关联：
- 电影基本信息（title_basics）
- 评分数据（title_ratings）
- 演职人员（name_basics）
- 剧组信息（title_crew）
- 剧集信息（title_episode）

PostgreSQL 在多表关联查询性能上优于 MySQL。

### 3. 数据导入性能

```sql
-- IMDb 数据导入使用 PostgreSQL 的 COPY 命令
\COPY title_basics FROM '/path/to/title.basics.tsv' WITH (FORMAT csv, DELIMITER E'\t', NULL '\N', HEADER true, QUOTE E'\b');
```

PostgreSQL 的 COPY 命令在大数据导入时性能极佳。

### 4. 数据分析需求

本项目需要复杂的数据分析：
- 题材分布统计
- 年代演变分析
- 评分分布分析
- 中西评分对比

PostgreSQL 的分析函数和聚合性能更适合此类场景。

## 四、MySQL 的局限性

### 1. JSON 处理能力弱
- MySQL 5.7+ 才支持 JSON，功能有限
- JSON 查询性能不如 PostgreSQL
- 不支持 JSONB（二进制存储）

### 2. 窗口函数支持晚
- MySQL 8.0+ 才支持窗口函数
- 功能不如 PostgreSQL 完善
- 性能优化较差

### 3. 并发控制问题
- MySQL InnoDB 的锁机制可能导致死锁
- 高并发写入时性能下降明显
- 间隙锁影响范围查询

### 4. 数据类型限制
- 不支持数组类型
- 不支持原生枚举类型（ENUM 实现方式不同）
- 几何类型支持有限

### 5. 扩展性差
- PostgreSQL 支持丰富的扩展（如 PostGIS、zhparser）
- MySQL 扩展生态相对薄弱

## 五、技术栈兼容性

### Python 生态
- **SQLAlchemy**：对 PostgreSQL 支持更完善
- **asyncpg**：PostgreSQL 专用异步驱动，性能极佳
- **psycopg2**：成熟的 PostgreSQL 同步驱动
- **Pandas**：read_sql 对 PostgreSQL 优化更好

### 大数据集成
- **PySpark**：对 PostgreSQL 的 JDBC 支持更稳定
- 数据导入导出性能更好

## 六、维护与运维优势

### 1. 更成熟的备份恢复
- PostgreSQL 的 pg_dump 工具功能强大
- 支持增量备份、时间点恢复（PITR）
- 本项目数据量大，备份恢复能力至关重要

### 2. 更好的监控工具
- pg_stat_statements 提供详细的查询统计
- 可用于性能优化和慢查询分析

### 3. 更活跃的开源社区
- PostgreSQL 是最先进的开源数据库
- 持续的性能优化和新特性
- 企业级支持（如 EnterpriseDB）

## 七、答辩要点

### 技术选型理由
1. **异步架构适配**：asyncpg 驱动性能优异，匹配 FastAPI 异步框架
2. **大数据处理**：千万级数据导入和查询性能更好
3. **复杂查询需求**：数据分析大屏需要复杂的聚合和窗口函数
4. **数据完整性**：严格的外键和约束保证数据质量
5. **扩展性**：支持 JSON、数组等高级数据类型

### 与 MySQL 对比
- PostgreSQL 在并发性能、复杂查询、数据分析方面全面优于 MySQL
- MySQL 更适合简单的 CRUD 应用，不适合本项目的大数据+分析场景
- PostgreSQL 的学习曲线稍陡，但长期收益更大

### 实际应用体现
- 数据导入：使用 COPY 命令快速导入千万级 IMDb 数据
- 性能优化：通过索引和查询优化实现毫秒级响应
- 数据分析：支持复杂的多维度统计和可视化
- 系统稳定性：高并发场景下表现稳定，无锁竞争问题

## 八、总结

PostgreSQL 是本项目数据库的最佳选择，其强大的功能、优秀的性能和成熟的生态完全满足项目需求。相比 MySQL，PostgreSQL 在并发控制、复杂查询、数据分析和扩展性方面具有明显优势，能够支撑本项目的长期发展和维护。

### 7. 数据模型定义（models.py）

```python
# PostgreSQL 特有的数据类型和约束
class User(Base):
    __tablename__ = "users"
    id = Column(Integer, primary_key=True, index=True)
    username = Column(String, unique=True, index=True)
    role = Column(String, default="user")

class UserFavorite(Base):
    __tablename__ = "user_favorites"
    # 联合唯一约束
    __table_args__ = (
        UniqueConstraint('user_id', 'tconst', name='uix_user_favorite'),
    )

class MovieBoxOffice(Base):
    __tablename__ = "movie_box_office"
    # 大整数类型存储票房
    box_office = Column(BigInteger, nullable=True)
    # 自动更新时间戳
    updated_at = Column(DateTime, default=datetime.utcnow, onupdate=datetime.utcnow)
```

**PostgreSQL 特性应用**：
- 联合唯一约束（防止重复收藏）
- BigInteger 类型（存储大数值）
- DateTime 自动更新
- 外键关联（ForeignKey）
- 索引自动创建

## 总结

在本项目中，PostgreSQL 的应用贯穿了整个系统架构：

1. **连接层**：asyncpg（异步）+ psycopg2（同步）双驱动支持
2. **数据层**：千万级 IMDb 数据的高效导入和存储
3. **业务层**：所有服务层的数据库操作（11 个服务文件）
4. **算法层**：Spark 大数据处理和推荐算法训练
5. **展示层**：数据可视化大屏的复杂查询支持
6. **运维层**：数据备份、恢复、性能监控

PostgreSQL 的强大功能（MVCC、复杂查询、JSON 支持、COPY 命令、UPSERT 等）在项目的各个层面都发挥了关键作用，是系统高性能、高可靠性的重要保障。
