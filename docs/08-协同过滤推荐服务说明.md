# 协同过滤推荐服务说明

## 1. 概述

`services/recommendation_service.py` 是项目的**实时协同过滤推荐服务**，采用 **Item-Based 协同过滤（基于物品的协同过滤）** 算法，利用用户的评分和收藏数据，通过余弦相似度计算电影之间的相似关系，为当前用户推荐其可能感兴趣的电影。

与 `spark_runner.py` 的离线 ALS 模型形成互补：
- **ALS 离线模型**：定期全量训练，生成全局 Top-10 推荐，覆盖全量用户
- **Item-Based 实时模型**：基于最新互动数据快速计算，响应更及时，适合前台实时调用

---

## 2. 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 算法 | **Item-Based 协同过滤** | 物品数量通常远少于用户数量，相似度矩阵更小、更稳定；适合电影推荐场景 |
| 相似度度量 | **余弦相似度 (Cosine Similarity)** | 衡量两部电影被哪些共同用户喜欢，对评分绝对值不敏感，适合稀疏评分矩阵 |
| 数据持久化 | **Pickle** | 模型为纯 Python DataFrame，Pickle 序列化/反序列化速度快，轻量 |
| 存储位置 | **本地磁盘 (`data/recommendation_model.pkl`)** | 服务启动时加载到内存，推荐查询零 IO 延迟 |
| ORM 交互 | **SQLAlchemy AsyncSession** | 复用项目统一的异步数据库会话 |

## 4. 核心模块

### 4.1 模型训练 (`train_model`)

```python
async def train_model()
```

**触发时机**：后台管理调用（如用户新增大量评分后手动触发，或定时任务触发）。

**数据准备**：
1. **评分数据**：从 `UserRating` 表读取 `user_id`、`tconst`、`rating`
2. **收藏数据**：从 `UserFavorite` 表读取 `user_id`、`tconst`，将收藏视为 **10 分满分评分**，以缓解数据稀疏问题
3. **去重归并**：若同一用户既评分又收藏同一部电影，取两者中的最高分

**矩阵构建**：
```python
pivot_matrix = df.pivot(index='tconst', columns='user_id', values='rating').fillna(0)
```

- 行 = 电影 (`tconst`)
- 列 = 用户 (`user_id`)
- 值 = 评分（未评分为 `0`）

**相似度计算**：
```python
cosine_sim = cosine_similarity(sparse_matrix)
```

对 `pivot_matrix` 的每两行（两部电影）计算余弦相似度，得到 N×N 的相似度矩阵（N 为电影数）。

**模型保存**：
```python
with open(MODEL_FILE, 'wb') as f:
    pickle.dump(_similarity_df, f)
```

文件路径：`项目根目录/data/recommendation_model.pkl`

### 4.3 实时推荐 (`get_recommendations`)

```python
async def get_recommendations(user_id: int, limit=8, category='all')
```

**核心推荐逻辑**（基于内存相似度矩阵）：

#### 步骤 1：获取用户种子电影

- 查询该用户评分最高的 **20 部电影**
- 查询该用户所有**收藏电影**
- 以评分/收藏分数作为**权重**（收藏 = 10.0 分）

#### 步骤 2：相似电影召回

对每部种子电影，从相似度矩阵中查找与其最相似的其他电影：
```python
similar_series = _similarity_df[watched_tconst]
similar_movies = similar_series[similar_series > 0.1]  # 过滤低相似度噪音
```

- **相似度阈值**：`> 0.1`，低于此值的相似电影被丢弃，减少噪音干扰

#### 步骤 3：加权聚合打分

```python
score = similarity * weight
candidate_scores[similar_tconst] += score
```

- 推荐分 = 电影相似度 × 用户对种子电影的喜爱权重
- 同一候选电影可能被多颗种子电影召回，分数累加
- **自动排除**用户已看过/收藏的电影

#### 步骤 4：排序、取 Top N、查详情

1. 按候选分数倒序取 **Top 100**
2. 查询 `MovieSummary` 表获取电影详情
3. 应用分类过滤（见 4.5 节）
4. 保持推荐顺序，截取前 `limit` 条返回

### 4.4 Spark 离线推荐查询 (`get_spark_recommendations`)

```python
async def get_spark_recommendations(user_id: int, limit=8, category='all')
```

该函数作为 **ALS 离线推荐的前端查询接口**，从 `spark_recommendations` 表中读取该用户的预计算推荐结果：

1. 按 `score` 降序取 **Top 100** `tconst`
2. 查询 `MovieSummary` 获取详情
3. 应用分类过滤
4. 保持推荐顺序，截取前 `limit` 条

此方式**无需实时计算**，响应速度极快，适合作为首页推荐的兜底或主数据源。

---

### 4.5 分类过滤 (`_apply_category_filter`)

推荐结果支持按影视类型进行二次过滤，过滤规则如下：

| 分类参数 | 说明 | 过滤条件 |
|----------|------|----------|
| `'all'` | 全部 | 不过滤 |
| `'movie'` | 电影 | `titleType` 为 `movie` 或 `tvMovie`，且排除 `Documentary` |
| `'tv'` | 电视剧 | `titleType` 为 `tvSeries` 或 `tvMiniSeries` |
| `'anime'` | 动画 | `genres` 包含 `Animation` |
| `'variety'` | 综艺 | `genres` 包含 `Reality-TV`、`Talk-Show` 或 `Game-Show` |
| `'doc'` | 纪录片 | `genres` 包含 `Documentary` |

> 分类过滤在 SQL 查询层完成，不破坏推荐排序，仅减少返回数量。

## 5. 数据流与协作关系

```
用户互动数据
    ├── 评分 -> UserRating 表
    └── 收藏 -> UserFavorite 表
            │
            ▼
    [train_model] Item-Based 模型训练
            │
            ├── 保存 -> data/recommendation_model.pkl
            │
            ▼
    [load_model] 启动加载到内存 _similarity_df
            │
            ├── [get_recommendations] 实时协同过滤推荐
            │
            └── [get_spark_recommendations] 读取 ALS 离线结果
                        │
                        ▼
                前台展示 (按 category 过滤)
```

## 7. 注意事项与优化建议

1. **冷启动处理**：新用户无评分/收藏时，`get_recommendations` 返回空列表，前台需降级到**热门推荐**或**新片榜单**
2. **数据稀疏性**：Item-Based 模型依赖共同评分用户；当评分数据极少时，余弦相似度可能失真，建议配合 ALS 离线模型作为双保险
3. **模型刷新频率**：用户新增评分后，建议触发 `train_model()` 重新训练，或设置定时任务（如每 6 小时/每天）后台刷新
4. **内存占用**：`_similarity_df` 为 N×N 矩阵，N 为电影数。若电影数超过 10 万，请考虑改用稀疏矩阵存储或切换为 ALS 方案
5. **扩展性**：当前为单进程内存计算，若需横向扩展，可将模型迁移至 Redis / 向量数据库，或使用 Spark 分布式推理
