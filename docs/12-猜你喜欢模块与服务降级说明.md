# 猜你喜欢模块与服务降级说明

## 1. 概述

"猜你喜欢"模块是影视推荐系统的核心功能，采用**三层推荐策略**，通过服务降级机制确保在任何情况下都能为用户提供推荐结果。该模块融合了离线计算、实时协同过滤和兜底策略，实现了高可用性与个性化的平衡。

---

## 2. 整体架构

### 2.1 三层推荐策略

```
用户请求推荐
    │
    ├─→ 第一层：Spark ALS 离线推荐（预计算结果）
    │       └─→ 有结果？返回 ✓
    │       └─→ 无结果？降级到第二层
    │
    ├─→ 第二层：Item-Based 实时协同过滤（内存计算）
    │       └─→ 有结果？返回 ✓
    │       └─→ 无结果？降级到第三层
    │
    └─→ 第三层：热门榜单兜底（按热度排序）
            └─→ 必有结果返回 ✓
```

### 2.2 核心组件

| 组件 | 文件 | 职责 |
|------|------|------|
| **离线推荐引擎** | `spark_runner.py` | 定期训练 ALS 模型，生成全量用户推荐 |
| **实时推荐服务** | `services/recommendation_service.py` | Item-Based 协同过滤，实时计算推荐 |
| **前端降级逻辑** | `pages/user_home.py` | 三层策略编排，确保高可用 |
| **推荐结果表** | `spark_recommendations` | 存储 ALS 离线推荐结果 |
| **模型文件** | `data/recommendation_model.pkl` | 持久化 Item-Based 相似度矩阵 |

## 3. 服务降级实现

### 3.1 降级触发条件

服务降级在以下场景自动触发：

1. **Spark 推荐无结果**
   - 用户是新用户，ALS 模型未为其生成推荐
   - ALS 离线任务未执行或执行失败
   - 推荐结果被分类过滤后为空

2. **实时协同过滤无结果**
   - 模型文件未加载（`data/recommendation_model.pkl` 不存在）
   - 用户无评分/收藏历史（冷启动用户）
   - 相似度矩阵中无匹配电影
   - 推荐结果被分类过滤后为空

3. **兜底策略**
   - 前两层均无结果时，自动降级到热门榜单
   - 热门榜单基于 `numVotes`（投票数）降序排序
   - 支持按分类过滤（电影/电视剧/动漫/综艺/纪录片）

### 3.2 前端降级代码实现

在 `pages/user_home.py` 的第 460-486 行：

```python
# 推荐策略：Spark -> 实时CF -> 热门
data_source = await recommendation_service.get_spark_recommendations(
    user_id, limit=8, category=cat_val
)
is_personalized = True

# 2. 实时协同过滤 (带过滤)

    data_source = await recommendation_service.get_recommendations(
        user_id, limit=8, category=cat_val
    )

# 3. 兜底策略：如果个性化推荐在这个分类下没结果，就取该分类的"热门榜单"
if not data_source:
    is_personalized = False
    top_raw = await movie_service.get_homepage_movies(
        page=1, page_size=8, category=cat_val
    )
    data_source = top_raw

# --- UI 渲染逻辑 ---
if is_personalized:
    ui.label('✨ 根据您的口味生成').classes('text-xs text-purple-500 q-mb-xs')
else:
    ui.label('🔥 热门榜单 (暂无个性化数据)').classes('text-xs text-orange-400 q-mb-xs')
```

### 3.3 降级策略的优势

| 维度 | 说明 |
|------|------|
| **高可用性** | 即使推荐系统全部失效，热门榜单仍能保证服务可用 |
| **用户体验** | 前两层个性化推荐失败时，用户仍能看到优质内容 |
| **性能优化** | 热门榜单查询直接走索引，响应速度极快 |
| **冷启动友好** | 新用户立即获得推荐，无需等待数据积累 |

## 4. Spark 离线推荐（ALS 协同过滤）

### 4.1 算法原理

**ALS（Alternating Least Squares，交替最小二乘法）** 是一种矩阵分解算法，用于解决协同过滤中的隐因子模型问题。

#### 核心思想

将用户-物品评分矩阵 $R$ 分解为两个低维矩阵：
- 用户矩阵 $U$（用户 × 隐因子维度）
- 物品矩阵 $V$（物品 × 隐因子维度）

使得 $R \approx U \times V^T$

#### 训练过程

1. 固定物品矩阵 $V$，通过最小二乘法求解用户矩阵 $U$
2. 固定用户矩阵 $U$，通过最小二乘法求解物品矩阵 $V$
3. 交替迭代，直到收敛

### 4.2 实现细节

#### 数据准备

```python
query = "SELECT user_id, tconst, rating FROM user_personal_ratings"
pdf_ratings = pd.read_sql(query, engine)
```

从 `user_personal_ratings` 表读取用户评分数据。

#### 特征编码

```python
string_indexer = StringIndexer(inputCol="tconst", outputCol="movie_idx")
indexer_model = string_indexer.fit(ratings_df)
indexed_ratings = indexer_model.transform(ratings_df)
```

ALS 要求物品 ID 为数值型，使用 `StringIndexer` 将 `tconst` 映射为连续整数索引。

#### 模型训练

```python
als = ALS(
    rank=4,              # 隐因子维度
    maxIter=15,          # 最大迭代次数
    regParam=0.5,        # L2 正则化系数
    userCol="user_id",
    itemCol="movie_idx",
    ratingCol="rating",
    coldStartStrategy="drop",  # 冷启动策略：丢弃未见过用户/物品
    nonnegative=True      # 限制隐因子非负
)
model = als.fit(training)
```

#### 模型评估

```python
evaluator = RegressionEvaluator(
    metricName="rmse",
    labelCol="rating",
    predictionCol="prediction"
)
rmse = evaluator.evaluate(predictions)
```

使用 **RMSE（均方根误差）** 评估模型性能：

$$
RMSE = \sqrt{\frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2}
$$

- **RMSE > 1.2**：提示指标偏高，建议调参
- **RMSE ≤ 1.2**：模型性能良好

#### 生成推荐

```python
user_recs = model.recommendForAllUsers(10)
```

为所有用户并行生成 Top-10 推荐，结果写入 `spark_recommendations` 表。

### 4.3 离线推荐的特点

| 特点 | 说明 |
|------|------|
| **计算密集** | 矩阵分解计算量大，适合离线批处理 |
| **全局优化** | 基于全量数据训练，推荐质量高 |
| **延迟高** | 需要定期刷新，非实时更新 |
| **可扩展** | 支持分布式计算，适合大规模数据 |

## 5. 实时协同过滤（Item-Based CF）

### 5.1 算法原理

**Item-Based 协同过滤** 基于物品相似度进行推荐，核心思想是：

> "喜欢电影 A 的用户，通常也会喜欢与 A 相似的电影 B"

#### 相似度计算

使用 **余弦相似度（Cosine Similarity）** 衡量两部电影之间的相似度：

$$
\text{similarity}(A, B) = \frac{A \cdot B}{\|A\| \|B\|}
$$

其中 $A$ 和 $B$ 是两部电影在用户空间中的评分向量。

### 5.2 实现细节

#### 模型训练

```python
# 1. 构建用户-物品评分矩阵
pivot_matrix = df.pivot(index='tconst', columns='user_id', values='rating').fillna(0)

# 2. 计算余弦相似度
cosine_sim = cosine_similarity(pivot_matrix.values)

# 3. 转为 DataFrame（方便索引）
_similarity_df = pd.DataFrame(cosine_sim, index=pivot_matrix.index, columns=pivot_matrix.index)

# 4. 持久化保存
with open(MODEL_FILE, 'wb') as f:
    pickle.dump(_similarity_df, f)
```

#### 数据增强

为缓解数据稀疏问题，将用户收藏视为 **10 分满分评分**：

```python
# 收藏数据合并（权重设为 10.0）
for f in fav_data:
    data_list.append({'user_id': f[0], 'tconst': f[1], 'rating': 10.0})

# 去重：如果用户既收藏又评分，取最高分
df = df.groupby(['user_id', 'tconst'], as_index=False)['rating'].max()
```

#### 实时推荐

```python
# 1. 获取用户种子电影（评分 > 6 或收藏）
watched_movies = {tconst: rating}  # {电影ID: 权重}

# 2. 对每部种子电影，查找相似电影
for watched_tconst, weight in watched_movies.items():
    similar_series = _similarity_df[watched_tconst]
    similar_movies = similar_series[similar_series > 0.1]  # 过滤低相似度噪音

    # 3. 加权聚合打分
    for similar_tconst, similarity in similar_movies.items():
        score = similarity * weight
        candidate_scores[similar_tconst] += score

# 4. 排序取 Top N
sorted_candidates = sorted(candidate_scores.items(), key=lambda x: x[1], reverse=True)
```

### 5.3 实时推荐的特点

| 特点 | 说明 |
|------|------|
| **响应快** | 相似度矩阵加载到内存，查询零 IO 延迟 |
| **实时性** | 基于最新用户行为，无需等待模型刷新 |
| **数据稀疏敏感** | 依赖共同评分用户，数据少时效果差 |
| **内存占用** | N×N 矩阵，N 为电影数，适合中小规模 |

## 6. 三层策略的协作关系

### 6.1 数据流向

```
用户互动数据（评分/收藏）
    │
    ├─→ [spark_runner.py] ALS 离线训练
    │       └─→ spark_recommendations 表
    │
    └─→ [recommendation_service.train_model] Item-Based 训练
            └─→ data/recommendation_model.pkl
                    │
                    ▼
            [recommendation_service.load_model] 启动加载
                    │
                    ├─→ get_spark_recommendations() 查询 ALS 结果
                    │
                    └─→ get_recommendations() 实时计算
                            │
                            ▼
                    [user_home.py] 三层降级策略
                            │
                            └─→ 前台展示
```

### 6.2 适用场景对比

| 场景 | 推荐策略 | 理由 |
|------|----------|------|
| **老用户 + 数据充足** | Spark ALS | 全局优化，推荐质量最高 |
| **老用户 + 刚有新互动** | Item-Based CF | 实时响应，立即体现新偏好 |
| **新用户 + 无历史** | 热门榜单 | 冷启动兜底，保证可用性 |
| **分类过滤后无结果** | 热门榜单 | 该分类下个性化数据不足 |
| **推荐系统故障** | 热门榜单 | 服务降级，确保高可用 |

### 6.3 性能对比

| 指标 | Spark ALS | Item-Based CF | 热门榜单 |
|------|-----------|---------------|----------|
| **查询延迟** | ~10ms（数据库查询） | ~5ms（内存计算） | ~5ms（数据库查询） |
| **训练耗时** | 数分钟（离线） | 数十秒（后台） | 无需训练 |
| **数据新鲜度** | 按刷新周期（天/小时） | 实时 | 实时 |
| **个性化程度** | 高 | 中 | 低 |
| **冷启动支持** | 差 | 差 | 优 |

## 7. 运维与监控

### 7.1 模型刷新策略

#### Spark ALS 离线模型

建议通过定时任务定期刷新：

- **Linux Cron**：
  ```bash
  # 每天凌晨 2 点执行
  0 2 * * * cd /path/to/movie && python spark_runner.py
  ```

- **Windows 计划任务**：
  创建任务，每日定时执行 `python spark_runner.py`

#### Item-Based 实时模型

在以下场景触发重新训练：

1. 用户评分/收藏数据量显著增长（如新增 1000+ 条）
2. 管理员手动触发（后台管理界面）
3. 定时任务（如每 6 小时）

```python
from services.recommendation_service import train_model

success, message = await train_model()
```

### 7.2 监控指标

| 指标 | 监控方式 | 告警阈值 |
|------|----------|----------|
| **Spark RMSE** | 日志输出 | > 1.2 |
| **推荐命中率** | 前端埋点 | < 80% |
| **模型加载状态** | 后台管理面板 | 加载失败 |
| **热门榜单降级率** | 日志统计 | > 30% |

### 7.3 故障排查

#### Spark 推荐无结果

- 检查 `spark_recommendations` 表是否有数据
- 检查 ALS 离线任务是否正常执行
- 查看日志中的 RMSE 指标

#### Item-Based 推荐无结果

- 检查 `data/recommendation_model.pkl` 文件是否存在
- 检查用户是否有评分/收藏历史
- 查看日志中的模型加载状态

#### 全部降级到热门榜单

- 检查数据库连接是否正常
- 检查 `movie_summary` 表是否有数据
- 检查分类过滤逻辑是否正确

## 8. 扩展与优化

### 8.1 算法优化方向

1. **Spark ALS 调参**
   - 增大 `rank` 提升表达能力
   - 调整 `regParam` 平衡拟合与过拟合
   - 增加 `maxIter` 充分收敛

2. **Item-Based 优化**
   - 使用稀疏矩阵存储降低内存占用
   - 引入时间衰减因子，近期行为权重更高
   - 结合内容特征（类型、导演等）混合推荐

3. **冷启动改进**
   - 基于用户注册信息（年龄、性别）做人口统计学推荐
   - 引入基于内容的推荐（电影标签匹配）
   - 新用户引导流程，主动收集偏好

### 8.2 架构扩展方向

1. **分布式部署**
   - Spark 集群模式，支持千万级用户/电影
   - 推荐服务微服务化，独立扩缩容

2. **缓存优化**
   - Redis 缓存热门推荐结果
   - 用户推荐结果缓存（TTL 1 小时）

3. **A/B 测试**
   - 不同推荐策略分流测试
   - 基于点击率、转化率优化权重

## 9. 总结

"猜你喜欢"模块通过 **三层推荐策略 + 服务降级机制**，实现了：

- **高可用性**：任何情况下都有推荐结果
- **个性化**：老用户享受精准推荐
- **实时性**：新互动立即体现
- **可扩展**：支持算法和架构升级

核心设计思想是：**优先个性化，兜底热门化，确保服务始终可用**。
